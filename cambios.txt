Se desarrolló el script maestro OrdenDeCreacion.sql para estructurar y documentar el despliegue de la base de datos desde cero. Al requerir el uso del Modo SQLCMD, este archivo automatiza la ejecución secuencial de los scripts individuales respetando la jerarquía de las dependencias debido a las claves foráneas. Esta implementación evita errores de referencias cruzadas.

Para reflejar fielmente el DER, se eliminó la tabla intermedia TieneHistorial. En su lugar, se modifico la tabla Historial, incorporando directamente en ella las claves foráneas correspondientes a Guardaparque y Parque.

Se agregó el campo HoraInicio a la clave primaria, quedando PRIMARY KEY(Actividad, FechaActividad, HoraInicio). Para permitir que una misma actividad (por ejemplo, una excursión) pueda dictarse en distintos horarios durante un mismo día. Con la clave anterior, se bloqueaba la inserción de un segundo turno en la misma fecha.

Se cambiaron los campos de texto de longitud variable (como DescripcionAct, Titulo, RazonEgreso) de CHAR a VARCHAR.
Se aumentó la capacidad del campo NombreParque (VARCHAR(100)). Existen parques nacionales reales con nombres extensos, los cuales habrían hecho fallar los procesos de inserción si el límite se mantenía en 30 caracteres.
Se agrego el esquema PnSPabm a asignarEspecialidad, desasignarEspecialidad, desasignarEspecialidades y reasignarEspecialidad.
Correcion de “IF( (@temporada != 'Verano') OR (@temporada != 'Invierno') OR (@temporada != 'Otoño') OR (@temporada != 'Primavera'))” por  “IF (@temporada NOT IN ('Verano', 'Invierno', 'Otoño', 'Primavera'))” en altahorario.
Correciones menores y de sintaxis.

Observaciones (Posibles cambios)
Asignación de guías en la tabla Actividad:
Actualmente, el campo guia es una clave foránea en la tabla Actividad. Se asume que una actividad genérica le pertenece a un único guía de forma estática. Probablemente se debería registrar qué guía participa en cada tour, mover esta asignación a la tabla HorarioActividad, la instancia específica del tour. Esto permitiría que distintos guías tomen diferentes turnos de la misma actividad.

Consideraciones sobre el Cifrado de Datos (Entrega 8):
El proyecto exige aplicar cifrado a los datos sensibles, El campo DNI en la tabla Persona seria el candidato para esto. Actualmente, DNI está definido como INT NOT NULL UNIQUE. Si más se cifra esta columna (lo que requieriria cambiar el tipo de dato a VARBINARY), hay que tener en cuenta que las funciones de cifrado de SQL Server pueden generar resultados diferentes para un mismo valor de entrada. Esto romperia la validación del constraint UNIQUE nativo, por lo que la validación de duplicados deberá manejarse con una estrategia distinta (por ejemplo, mediante una columna de hash complementaria o directamente en la lógica del procedimiento almacenado).
